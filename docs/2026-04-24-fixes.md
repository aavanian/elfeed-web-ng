## Architectural notes

### Long-poll race condition (issue 1)

`elfeed/feed-update-done` is a long-poll endpoint: the browser hits it and waits; the server parks the HTTP process on `elfeed-web-ng-feed-done-waiting` and drains the list when the queue empties.

**The race:** if the feed update finishes *before* the browser issues the `feed-update-done` request, `check-queue-done` fires, finds no waiters, and stops. The browser's proc is then parked with nothing to ever drain it → infinite hang.

**The fix:** always check `(zerop (elfeed-queue-count-total))` at the top of the handler. In single-threaded Emacs this check-then-park is atomic with respect to the timer. If already idle, respond immediately; otherwise park as before.

**General principle:** any long-poll design must handle the case where the condition is satisfied before the client registers. The safe pattern is check-then-park, exploiting the single-threaded execution model.

### `intern` and the Emacs obarray (issue 2)

`(intern str)` creates a symbol in the global `obarray` and retains it for the life of the Emacs process. There is no way to remove a symbol from the default obarray. Any API that accepts arbitrary strings and converts them to symbols via `intern` is therefore a memory sink.

Elfeed requires symbols for its tag API (e.g., `elfeed-tag` accepts symbols). The fix is to validate the string *before* interning: enforce a maximum length and an allowlist of characters (`[a-zA-Z0-9_★\-]`). Invalid tags are rejected with a 400 rather than silently interned.

### webid birthday collision (issue 4)

webids are 8 base64 chars of a SHA1 hash = 48 bits of effective entropy.

Birthday collision probability with N items: approximately `N² / 2^(bits+1)`.

| N (entries) | 48-bit (old) | 72-bit (new, 12 chars) |
|-------------|-------------|----------------------|
| 512 | ~1 in 500B | ~1 in 10¹⁷ |
| 10,000 | ~1 in 1.4B | negligible |

A collision is silent: `elfeed-web-ng-lookup` returns the first entry that registered the webid, so a tag update silently targets the wrong entry. 12 chars makes this practically impossible.

`elfeed-web-ng-webid-map` is session-scoped (populated on demand by `elfeed-web-ng--make-webid`). New webids are generated after a restart or `elfeed-web-ng-reset`, so lengthening the substring takes effect without migration.

### Content endpoint CSP (issue 3)

The iframe in `EntryContent` uses `srcdoc` + `sandbox="allow-popups"`, which is the primary defence against active content in feed HTML. The server-side CSP is a second line of defence:

```
Content-Security-Policy: sandbox allow-popups; default-src 'self'
```

This ensures the endpoint is also safe if loaded as a top-level URL (direct navigation, browser bug, future code regression). The `sandbox` directive in CSP is equivalent to the iframe `sandbox` attribute applied at the HTTP level.

### ARIA listbox pattern (issue 12)

`aria-selected` is only valid on elements with role `option`, `gridcell`, `row`, `tab`, or `treeitem`. A plain `<ul>/<li>` pair must not use `aria-selected`.

Choices:
1. **`role="listbox"` + `role="option"`** — semantically correct; screen readers announce the list as a selection widget and the selected option. Chosen here.
2. **`aria-current="true"`** — valid on any element; communicates "current item" (e.g., current page in a nav) rather than "selected option". Weaker semantics for a selection list.

The `role="listbox"` approach is correct but adds keyboard navigation expectations (arrow keys to move selection). Since the app is touch-first and keyboard nav is not implemented, this is an acknowledged trade-off.

### Annotation store sync (issue 15)

After `setAnnotation` succeeds, three places hold the entry data:
1. `store.entries` (the full list)
2. `store.selectedEntry` (the currently displayed entry)
3. `AnnotationEditor`'s local `text` state

Before the fix, only (3) was updated. The `Add annotation / Edit annotation` toggle button in `AnnotationEditor` reads `entry.annotation` from props, which comes from (2), which is stale until the next search.

The fix mirrors the `TagActions` → `EntryContent` → store update pattern:
- `AnnotationEditor` receives an `onAnnotationChanged(updatedEntry)` callback
- `EntryContent` implements `handleAnnotationChanged` that updates both (1) and (2)

`getAnnotation` in `api.js` exists but is never called. It is correct API surface for a future "always-fresh annotation on entry open" feature and is intentionally left in place.
